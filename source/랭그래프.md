 Part 1. LangGraph 기초
본 파트는 LangGraph의 근간이 되는 개념을 정리함. StateGraph의 3대 요소(상태·노드·엣지)를 중심으로 그래프 구성 원리를 이해하고, 메모리·서브그래프·스트리밍·Functional API까지 확장하여 실무 활용의 기반을 다짐.
1-1. LangGraph 개요
LangGraph의 정의
LangGraph는 LangChain 팀이 개발한 그래프 기반 LLM 오케스트레이션 프레임워크임. 상태(State)를 공유하는 노드(Node)들을 엣지(Edge)로 연결하여, 복잡한 워크플로우를 명시적이고 제어 가능한 형태로 표현함.
LangChain과의 관계
구분	LangChain	LangGraph
구조	선형 체인 (LCEL)	그래프 (사이클 허용)
흐름 제어	순차 처리	조건부 분기·반복·병렬
상태 관리	제한적 (RunnableWithMessageHistory)	명시적 State 스키마
적합 용도	단순 파이프라인, 프롬프트 체인	에이전트, RAG, 다중 에이전트
디버깅	체인 단위 로깅	노드·엣지 단위 추적 (LangSmith)

LangGraph를 선택하는 이유
•	사이클(반복) 지원: ReAct 에이전트의 Think-Act-Observe 루프를 자연스럽게 표현함.
•	명시적 상태 관리: TypedDict 기반 스키마로 상태 변화를 타입 안전하게 추적함.
•	체크포인팅: 그래프 실행의 각 단계를 저장하여 재개·롤백·Time Travel을 지원함.
•	Human-in-the-Loop: interrupt()로 사용자 승인·수정 단계를 매끄럽게 삽입함.
•	관찰 가능성: LangSmith·Studio 연동을 통한 시각화·디버깅이 원활함.
LangGraph 첫걸음 — 4단계 학습 흐름
•	1단계 — State 정의: TypedDict로 그래프에서 공유할 데이터 구조를 명세함.
•	2단계 — Node 함수 작성: 상태를 입력받아 갱신된 상태를 반환하는 순수 함수 형태로 정의함.
•	3단계 — Graph 조립: StateGraph에 노드와 엣지(START → 노드 → END)를 등록함.
•	4단계 — 실행: compile() 후 invoke() 또는 stream()으로 그래프를 구동함.
[예시] LangGraph 최소 코드
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 1단계: 상태 정의
class State(TypedDict):
    counter: int

# 2단계: 노드 함수
def increment(state: State) -> State:
    return {"counter": state["counter"] + 1}

# 3단계: 그래프 조립
graph = StateGraph(State)
graph.add_node("inc", increment)
graph.add_edge(START, "inc")
graph.add_edge("inc", END)

# 4단계: 실행
app = graph.compile()
result = app.invoke({"counter": 0})
# result == {"counter": 1}

 1-3. StateGraph 이해
StateGraph는 LangGraph의 핵심 클래스로, 상태 스키마와 노드·엣지의 집합으로 그래프를 정의함. 본 절에서는 3대 구성 요소인 State, Node, Edge를 개념적으로 정리함.
1-3-1. 기본 구성 요소 개관
요소	역할	구현 방식
State	그래프에서 공유되는 데이터 구조	TypedDict / Pydantic
Node	상태를 변환하는 작업 단위	Python 함수 (sync/async)
Edge	노드 간 실행 순서·조건 정의	add_edge / add_conditional_edges

1-3-2. 상태 (State)
•	정의: 그래프 실행 전 과정에서 공유되는 데이터 컨테이너임.
•	선언: TypedDict, Pydantic BaseModel, dataclass 중 선택 가능함.
•	갱신 원칙: 노드는 상태의 부분 갱신(dict)을 반환하며, LangGraph가 병합함.
리듀서(Reducer) — 상태 병합 전략
리듀서는 동일 키에 대해 여러 노드가 값을 반환할 때 병합 방식을 정의하는 함수임. Annotated 타입 힌트로 지정하며, 리스트 병합·메시지 append·집합 합집합 등에 활용함.
리듀서	동작	대표 용도
기본(덮어쓰기)	마지막 값이 이전 값을 대체함	카운터, 플래그 등 단일 값
operator.add	리스트/문자열/숫자 연결·합산	누적 로그, 총합
add_messages	메시지 리스트 append 및 중복 제거	대화 히스토리 (MessagesState)
set/union 커스텀	집합 합집합, 정렬 병합 등	고급 커스터마이징

MessagesState — 대화형 앱의 표준
MessagesState는 messages: list[AnyMessage] 필드를 자동 관리하는 사전 정의 클래스임. add_messages 리듀서를 내장하여, 챗봇·에이전트의 대화 흐름을 간결하게 표현함.
Private State — 노드 간 비공개 데이터
일부 노드만 사용하는 중간 데이터는 별도 스키마(Private State)로 분리 가능함. 그래프 전역 상태를 오염시키지 않으면서, 특정 구간에서만 필요한 정보(중간 계산 결과·디버그 데이터)를 안전하게 관리함.
1-3-3. 노드 (Node)
•	정의: State를 입력받아 갱신된 State(부분 dict)를 반환하는 함수임.
•	시그니처: def node(state: State, config: RunnableConfig = None) -> dict 형태를 취함.
•	동기·비동기: async def로도 정의 가능하며, LangGraph가 자동으로 조율함.
•	특수 노드: START, END는 진입점·종료점을 나타내는 예약 상수임.
•	캐싱: 동일 입력에 대해 캐시하여 재실행 비용을 절감할 수 있음(node caching).
노드 설계 원칙
•	순수성: 사이드 이펙트(파일 I/O·외부 호출)는 최소화하고, 명시적으로 격리함.
•	단일 책임: 하나의 노드는 하나의 논리적 작업만 수행함.
•	부분 반환: 전체 State가 아닌 변경된 키만 dict로 반환함으로써 병합 부담을 줄임.
•	구성 활용: RunnableConfig의 configurable 필드로 실행별 파라미터를 주입함.
1-3-4. 엣지 (Edge)
•	정의: 노드 간 실행 순서 및 조건을 명시하는 연결 규칙임.
•	종류: 정적 엣지(add_edge), 조건부 엣지(add_conditional_edges), Command 반환 기반 동적 엣지.
•	병렬 실행: 하나의 노드에서 다수의 노드로 fan-out 가능하며, 결과는 리듀서로 병합됨.
조건부 엣지 (Conditional Edges)
라우터 함수가 State를 검사하여 다음에 실행할 노드명(또는 END)을 문자열로 반환함. ReAct 에이전트의 ‘도구 호출 여부에 따른 분기’가 대표적 활용 예시임.
[예시] 조건부 라우팅
def route(state: MessagesState) -> str:
    last = state["messages"][-1]
    if last.tool_calls:
        return "tools"
    return END

graph.add_conditional_edges("agent", route, {"tools": "tools", END: END})


Command — 상태 갱신과 라우팅 결합
노드 함수가 Command 객체를 반환하면 상태 갱신(update)과 다음 노드 지정(goto)을 동시에 수행함. 복잡한 흐름을 노드 내부에서 자기완결적으로 표현하는 데 유용함.
Send API — 동적 팬아웃
런타임 데이터에 따라 동일 노드를 여러 번, 서로 다른 입력으로 병렬 호출하는 API임. map-reduce 패턴, 배치 문서 처리, 다중 도구 병렬 실행에 적합함.
1-3-5. 그래프 컴파일 및 실행
메서드	특징	적합한 상황
invoke	전체 실행 후 최종 상태 반환	짧고 결과만 필요한 작업
stream	노드별 중간 결과 실시간 반환	UI 표시, 진행 상황 모니터링
astream	비동기 스트리밍	고동시성 서버 환경
batch	다중 입력 병렬 처리	대량 배치 처리

Part 1 핵심 요약 (1-1~1-3)
LangGraph = TypedDict 기반 State + 순수 함수 Node + 정적·조건부·Command·Send 엣지. 이 4가지 요소만으로 ReAct·RAG·다중 에이전트 등 대부분의 LLM 워크플로우를 표현 가능함.

 1-4. 메모리 (Memory)
메모리는 그래프 실행 사이의 상태를 보존하는 메커니즘임. LangGraph는 2단계 메모리(단기·장기)를 명확히 구분하여, 대화 지속성부터 사용자별 개인화 데이터까지 폭넓게 지원함.
2단계 메모리 시스템
구분	단기 메모리	장기 메모리
명칭	Checkpointer	Store
보존 단위	thread_id 별 대화 세션	사용자·조직 등 전역 네임스페이스
보존 기간	세션 종료 시점까지	영구 (명시적 삭제 시까지)
대표 용도	대화 이력, Time Travel, 재개	사용자 프로필, 선호 설정, 지식
구현체	InMemorySaver, SqliteSaver, PostgresSaver	InMemoryStore, PostgresStore

1-4-1. 메모리 기본 예제
•	체크포인터는 compile(checkpointer=...) 인자로 그래프에 부착함.
•	실행 시 config={"configurable": {"thread_id": "user-1"}} 형태로 세션 구분함.
•	동일 thread_id로 재호출 시 이전 상태에서 자동으로 이어짐.
[예시] Checkpointer 부착 및 thread 구분
from langgraph.checkpoint.memory import InMemorySaver

app = graph.compile(checkpointer=InMemorySaver())

config = {"configurable": {"thread_id": "user-1"}}
app.invoke({"messages": [("user", "안녕")]},    config=config)
app.invoke({"messages": [("user", "이름은?")]}, config=config)  # 이전 대화 기억


1-4-2. 체크포인터 종류별 선택 가이드
체크포인터	저장소	적합한 환경
InMemorySaver	프로세스 메모리	개발·테스트, 단발성 실험
SqliteSaver	SQLite 파일	로컬 앱, 소규모 프로토타입
PostgresSaver	PostgreSQL DB	프로덕션, 다중 인스턴스
Redis 커뮤니티	Redis	고성능·저지연 시나리오

1-4-3. Time Travel — 과거 상태로 이동
•	개념: 체크포인터에 저장된 과거 시점의 상태에서 실행을 분기(fork)함으로써, ‘만약 그 시점부터 다르게 실행했다면?’을 실험함.
•	필요성: 대화형 앱에서 사용자가 이전 답변을 되돌리고 다른 방향을 시도하는 시나리오 지원.
•	구현: get_state_history()로 과거 체크포인트 조회 → 원하는 상태에서 update_state() → invoke()로 재개함.
•	활용 패턴: 번역 품질 개선, 프롬프트 A/B 실험, 사용자 되돌리기(Undo) 기능.
Time Travel vs 일반 재실행
일반 재실행은 처음부터 다시 계산하지만, Time Travel은 저장된 중간 상태에서 시작하므로 비용·시간을 크게 절감함. 사이클이 많거나 LLM 호출이 무거운 그래프에서 특히 강력함.

1-4-4. Durable Execution — 견고한 실행
•	개념: 노드 단위 결과가 체크포인터에 저장됨으로써, 프로세스 중단 후에도 마지막 성공 지점부터 재개 가능한 실행 방식임.
•	필요성: 장시간 실행 워크플로우, 외부 API 호출 실패, 서버 재시작 등 장애 대응에 필수적임.
•	리플레이 메커니즘: 재실행 시 이미 완료된 노드는 캐시된 결과를 사용하고, 실패 지점 이후만 재수행함.
•	Durability Modes: exit(기본, 노드 완료 시 저장), async(비동기 저장), sync(즉시 저장) 등 성능·안전성 트레이드오프 조절 가능.
•	Determinism 요구: 노드는 결정론적이어야 리플레이가 안전함. 랜덤·시간·외부 상태는 상태에 저장하거나 결정론적으로 시딩함.
 1-5. 서브그래프 (Sub-graph)
서브그래프는 그래프를 노드처럼 다른 그래프에 삽입하는 모듈화 기법임. 복잡한 시스템을 재사용 가능한 단위로 분해함으로써 유지보수성과 테스트 용이성을 높임.
서브그래프 활용 목적
•	모듈화: 기능 단위(예: RAG 파이프라인)를 독립적으로 개발·테스트함.
•	재사용: 하나의 서브그래프를 여러 상위 그래프에서 공유함.
•	복잡도 관리: 큰 그래프를 여러 계층으로 나누어 가독성 확보.
•	다중 에이전트: 각 에이전트를 서브그래프로 표현하여 계층 구조를 명확화함.
추가 방법 2가지
방법	상태 스키마	특징
add_node로 직접 추가	부모와 스키마 공유	간편함, 상태 자동 전달
노드 함수에서 invoke 호출	다른 스키마 사용 가능	스키마 변환 필요, 유연함

스트리밍과 상태 조회
•	서브그래프 실행 이벤트를 부모 스트림에서 관찰하려면 subgraphs=True 옵션 사용함.
•	체크포인터는 부모에서 자동 전파되어 서브그래프에도 동일하게 적용됨.
•	get_state()로 서브그래프 내부 상태를 조회할 수 있음(경로 지정 필요).
설계 원칙
•	인터페이스 명확화: 서브그래프의 입력·출력 스키마를 상위와 명확히 분리 계약함.
•	책임 분리: 하나의 서브그래프는 하나의 명확한 목적만 수행함.
•	테스트 가능성: 서브그래프 단독 실행 테스트를 우선함.
1-6. 스트리밍 (Streaming)
스트리밍은 그래프 실행 중간의 상태·토큰·커스텀 이벤트를 실시간으로 수신하는 기능임. 사용자 경험 향상, 디버깅, 실시간 UI 업데이트에 필수적임.
Stream Mode 4종
모드	출력 내용	대표 활용
values	각 단계의 전체 상태 스냅샷	상태 진화 관찰, 디버깅
updates	각 노드가 반환한 변경분만	노드별 활동 추적, 로그
messages	LLM 토큰 단위 출력	실시간 채팅 UI
custom	사용자 정의 이벤트	진행률 표시, 사이드 이벤트

다중 모드 동시 사용
stream_mode=["updates", "messages"]와 같이 리스트로 지정하면 여러 이벤트 종류를 하나의 스트림에서 병합하여 수신 가능함. 실무에서는 UI에는 messages, 로깅에는 updates를 병행 사용하는 패턴이 일반적임.
토큰 스트리밍 필터링
•	노드별 필터링: metadata의 langgraph_node로 특정 노드의 토큰만 선택함.
•	태그 기반 필터링: LLM 호출 시 with_config(tags=[...])로 붙인 태그로 구분함.
•	멀티 모델: 여러 LLM을 사용할 때 각 모델의 출력을 태그·노드로 분리함.
커스텀 스트리밍
get_stream_writer()를 노드·도구 내부에서 호출하여 임의의 이벤트(진행률, 중간 계산 결과 등)를 전송함. 특히 장시간 실행 작업에서 진행 표시줄(progress bar) 구현에 유용함.
 1-7. Functional API
Functional API는 Graph API의 대안으로, @entrypoint와 @task 데코레이터로 워크플로우를 함수형 코드처럼 작성하는 API임. 그래프 구조를 명시적으로 그리지 않아도 자연스러운 파이썬 함수 흐름으로 표현 가능함.
핵심 구성 요소
데코레이터	역할	특징
@entrypoint	워크플로우 진입점 함수 지정	체크포인터·설정 통합
@task	재사용 가능한 하위 작업 함수	결과 캐싱, 병렬 실행 지원

Graph API vs Functional API
기준	Graph API	Functional API
표현 방식	그래프 구조 명시적 정의	함수 호출 흐름으로 자연스럽게 표현
시각화	우수 (Mermaid, Studio)	제한적
복잡한 분기	명시적 조건부 엣지	if/for 등 파이썬 구문
학습 곡선	다소 가파름	완만함 (Python에 익숙하면 즉시 사용)
적합 용도	명확한 상태 머신, 다중 에이전트	선형·간단한 조건 흐름

API 선택 기준
•	그래프 구조·상태 다이어그램이 중요하면 Graph API 선택함.
•	코드 가독성과 함수형 스타일을 우선하면 Functional API 선택함.
•	하이브리드: 상위는 Functional로 오케스트레이션, 하위는 Graph로 세부 상태 관리 가능함.
Part 1 마무리
Part 1은 LangGraph 사용의 전 과정을 개괄함. State–Node–Edge 3요소 위에 메모리·서브그래프·스트리밍·Functional API를 얹어 프로덕션급 애플리케이션 구축이 가능함. Part 2부터는 이 기초 위에 실전 패턴을 쌓아 올림.

 Part 2. ReAct 에이전트 구현
본 파트는 ReAct 패턴 기반의 에이전트 설계를 다룸. 고수준 create_agent와 저수준 StateGraph의 두 축을 비교하고, 미들웨어·Human-in-the-Loop·가드레일 등 실무 요소를 통합함.
2-1. ReAct 에이전트 개요
ReAct 패러다임의 핵심
ReAct는 Reasoning + Acting의 합성어로, LLM이 ‘생각(Think) → 도구 사용(Act) → 관찰(Observe)’ 사이클을 반복하며 복잡한 문제를 해결하는 에이전트 패턴임. 단순 Chain-of-Thought보다 외부 도구를 통한 검증·정보 확보에서 강점을 가짐.
Think → Act → Observe 사이클
•	Think: LLM이 현재 상황을 분석하고 다음 행동을 계획함.
•	Act: 선택한 도구를 호출하여 외부 세계와 상호작용함.
•	Observe: 도구 실행 결과를 관찰하고 상태를 갱신함.
•	반복: 목표 달성 또는 종료 조건 충족 시까지 사이클을 지속함.
Chain-of-Thought vs ReAct
항목	Chain-of-Thought	ReAct
외부 세계 상호작용	없음 (모델 지식만 사용)	도구로 실시간 정보 획득
신뢰성	환각(hallucination) 가능	도구 결과로 검증됨
투명성	추론 과정 기록	추론 + 행동 로그 병행
대표 용도	논리·수학 추론	검색, 계산, 코드 실행, 툴 사용

ReAct의 주요 장점
•	투명성: 각 추론 단계를 명시적으로 추적함.
•	신뢰성: 외부 도구를 통한 사실 검증으로 환각을 억제함.
•	확장성: 새로운 도구를 추가하여 능력을 유연하게 확장함.
•	제어성: 각 단계에 인간 개입 지점을 삽입할 수 있음.
LangGraph에서의 ReAct 구현 방법
구분	create_agent (고수준)	StateGraph (저수준)
코드량	3~5줄로 완성	노드·엣지 직접 정의
커스터마이징	미들웨어로 확장	완전 자유
적합 상황	표준 ReAct, 빠른 프로토타입	복잡한 커스텀 흐름
학습 우선순위	먼저 익힘	필요 시 심화

recursion_limit — 무한 루프 방지
에이전트가 종료 조건을 찾지 못하고 반복될 위험이 있으므로, recursion_limit(기본 25)으로 최대 반복 횟수를 제한함. 간단 질의는 10~15, 중간 복잡도는 25~50, 복잡한 리서치는 50~100 수준으로 조정함.
도구 통합 아키텍처 — Function Calling
•	@tool 데코레이터: 파이썬 함수를 LLM이 호출 가능한 도구로 변환함.
•	bind_tools: LLM에 도구의 JSON Schema를 전달하여 자동 호출을 활성화함.
•	ToolNode: LLM이 반환한 tool_calls를 실제로 실행하여 결과를 상태에 추가함.
•	라우팅 함수: 마지막 메시지에 tool_calls가 있으면 tools 노드로, 없으면 END로 분기함.
[예시] 저수준 StateGraph 기반 ReAct 골격
from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.prebuilt import ToolNode
from langchain_core.tools import tool

@tool
def calculator(expr: str) -> str:
    """수식을 계산함."""
    return str(eval(expr))

llm_with_tools = llm.bind_tools([calculator])

def agent_node(state: MessagesState):
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

def route(state):
    return "tools" if state["messages"][-1].tool_calls else END

g = StateGraph(MessagesState)
g.add_node("agent", agent_node)
g.add_node("tools", ToolNode([calculator]))
g.add_edge(START, "agent")
g.add_conditional_edges("agent", route, {"tools": "tools", END: END})
g.add_edge("tools", "agent")
app = g.compile()


 2-2. create_agent 고수준 API
create_agent는 표준 ReAct 그래프(agent ↔ tools 루프)를 자동으로 생성하는 팩토리 함수임. 도구·모델·시스템 프롬프트만 지정하면 프로덕션 수준의 에이전트를 즉시 얻을 수 있음.
커스텀 도구 정의 원칙
•	명확한 이름과 상세한 docstring 필수 — LLM이 도구 선택 근거로 사용함.
•	타입 힌트로 파라미터 스키마 명시 — 자동 검증에 활용됨.
•	부수 효과 최소화, 결과 문자열은 요약된 형태로 반환함.
모델 초기화 · 에이전트 생성 · 실행
[예시] create_agent 표준 사용
from langchain.chat_models import init_chat_model
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

model  = init_chat_model("openai:gpt-4.1-mini", temperature=0)
tools  = [calculator, tavily_search]
agent  = create_agent(model, tools, checkpointer=InMemorySaver())

config = {"configurable": {"thread_id": "chat-1"}}
agent.invoke({"messages": [("user", "2^10 계산해줘")]}, config=config)


미들웨어 시스템
미들웨어는 에이전트 실행 흐름에 횡단 관심사(로깅·PII 마스킹·에러 처리·HITL 등)를 삽입하는 확장 지점임. 표준 훅(before_model, after_model, before_tool 등)에 함수를 등록함으로써 코어 로직을 수정하지 않고 능력을 확장 가능함.
미들웨어 종류	역할
PII 마스킹	LLM 호출 전 이메일·전화번호를 [REDACTED]로 대체함
대화 요약	긴 히스토리를 자동 요약하여 컨텍스트 절약
HumanInTheLoop	민감 도구 호출 전 사용자 승인 요구
에러 처리	도구 실행 실패를 캐치하여 재시도·폴백 수행
비용 추적	토큰·호출 수 계측

스트리밍과 디버깅
•	stream(): 노드별 실행 결과와 도구 호출 이벤트를 실시간 수신함.
•	LangSmith 연동: 프로덕션 트레이스·평가·회귀 감지에 필수적임.
•	recursion_limit: 무한 루프 방지의 기본 안전장치임.
 2-3. StateGraph 커스텀 에이전트
직접 구현이 필요한 경우
•	표준 ReAct 루프에서 벗어난 커스텀 흐름(다단계 승인, 병렬 도구 실행, 복합 라우팅)이 필요할 때 사용함.
•	State에 messages 이외의 도메인 필드를 다수 추가할 때 유리함.
•	미들웨어로 표현하기 어려운 복잡한 제어가 필요할 때 채택함.
핵심 구성요소 정리
•	MessagesState 확장: messages + 사용자 정의 필드를 함께 관리함.
•	agent 노드: LLM.bind_tools() 결과를 호출하고 응답 메시지를 추가함.
•	tools 노드: ToolNode 사용으로 tool_calls를 자동 실행함.
•	라우터: tool_calls 존재 여부에 따라 tools 또는 END로 분기함.
•	체크포인터: InMemorySaver/SqliteSaver 등을 부착하여 대화 지속성 확보.
에러 처리와 재시도 전략
•	도구 실행 실패를 try/except로 포착하고, 재시도 정책(횟수·백오프)을 적용함.
•	지속 실패 시 대체 도구·안전한 응답으로 폴백함.
•	실패 로그를 상태에 남겨 이후 노드가 회복 전략을 결정하도록 함.
create_agent vs StateGraph 선택 기준
표준 ReAct + 미들웨어로 충분하면 create_agent가 명료함. 복잡한 도메인 상태·비표준 흐름·다단계 제어가 필요하면 StateGraph가 유리함. 둘은 상호 배타적이지 않으며, 프로젝트 진행에 따라 이전 가능함.

 2-4. Human-in-the-Loop (HITL)
HITL은 자동화된 에이전트 흐름 사이에 사람의 판단·승인·수정을 삽입하는 메커니즘임. 고위험 작업(결제·이메일 발송·파일 삭제) 승인, 애매한 결정 확인, 품질 검토 등에 필수적임.
interrupt()의 기본 메커니즘
•	노드 내부에서 interrupt(payload)를 호출하면 그래프 실행이 중단되고 payload가 클라이언트로 반환됨.
•	사람이 결정 값을 Command(resume=...)로 전달하면 정확히 같은 지점에서 재개됨.
•	Checkpointer가 필수 — 중단 시점의 상태를 저장해야 재개 가능함.
구현 방법 비교
방법	특징	적합한 상황
StateGraph + interrupt()	완전 커스텀, 조건부 라우팅 자유	복잡한 승인 흐름, 다단계 결재
HumanInTheLoopMiddleware	create_agent에 미들웨어로 통합	표준 도구 승인, 빠른 도입

Interrupt 4대 규칙
•	1) 결정론성 유지: interrupt 호출 순서가 실행마다 동일해야 함.
•	2) 직렬화 가능 값만 전달: dict, list, str, int, bool, None 만 payload로 허용함.
•	3) 부수 효과는 interrupt 이후: 재개 시 중복 실행 방지를 위함.
•	4) 재개 값도 직렬화 가능: Command(resume=...)에 함수·객체 전달 금지.
고급 패턴
•	조건부 라우팅: 승인/거부/수정/에스컬레이션 결과에 따라 다음 노드를 다르게 지정함.
•	다단계 승인: 1차 승인 → 2차 승인 등 결재 체계 구현.
•	타임아웃 처리: 승인 대기 시간이 지나면 자동 거부 또는 알림 발송.
2-5. Guardrails & Safety
가드레일은 에이전트의 입력·출력·행동을 안전 범위 내로 제한하는 방어 계층임. 결정론적·모델 기반·HITL 세 계층을 조합함으로써 견고한 안전성을 확보함.
2-5-1. 결정론적 가드레일
•	PII 마스킹: 정규식으로 이메일·전화번호·주민번호를 사전에 치환함.
•	Prompt Injection 방어: 특정 문구·패턴을 블록리스트로 사전 차단함.
•	특징: 예측 가능·저비용·저지연이지만, 새로운 우회 시나리오에는 취약함.
2-5-2. 모델 기반 가드레일 (LLM-as-Judge)
•	별도 LLM이 입력·출력을 검사하여 정책 위반 여부를 분류함.
•	정교하고 확장 가능하지만, 지연·비용·판정 편차의 트레이드오프가 있음.
•	실무 팁: 소형 저비용 모델 + 명확한 rubric 프롬프트로 비용을 통제함.
2-5-3. HITL 가드레일
고위험 도구 호출 직전 interrupt로 사람 승인 요구. 결정론·모델 판정이 애매한 상황에서 최종 안전망 역할 수행.
2-5-4. 통합 파이프라인
실무 표준: (1) 결정론적 필터로 명백한 위험 즉시 차단 → (2) 모델 기반 판정으로 미묘한 위반 감지 → (3) HITL로 애매한 사례 최종 확인. 세 계층을 조합하여 정확도·비용·안전성을 균형 있게 확보함.
 Part 3. RAG(Retrieval-Augmented Generation) 시스템 구축
본 파트는 LLM의 지식 한계를 외부 문서 검색으로 보완하는 RAG 시스템 구축을 다룸. LCEL 체인 방식과 StateGraph 방식의 차이를 이해하고, 문서 처리·벡터 DB·평가까지 전 과정을 정리함.
3-1. RAG 개요와 아키텍처
RAG의 등장 배경
•	LLM은 학습 시점 이후 정보에 접근 불가하며, 특정 도메인 지식이 부족함.
•	파인튜닝은 비용·시간이 크고, 지식 업데이트가 유연하지 않음.
•	RAG는 외부 문서를 실시간 검색하여 최신·전문 지식을 프롬프트에 주입함.
RAG의 3단계 워크플로우
•	① Ingestion(사전 준비): 문서 로드 → 청킹 → 임베딩 → 벡터 DB 저장.
•	② Retrieval(질의 시): 사용자 질문을 임베딩하고 유사 청크를 검색함.
•	③ Generation(응답 생성): 검색된 컨텍스트를 프롬프트에 삽입하여 LLM이 답변 생성.
LCEL 체인 vs StateGraph RAG
항목	LCEL 체인	StateGraph RAG
표현 방식	선형 파이프 (retriever | prompt | llm)	노드·엣지 기반 그래프
조건부 분기	제한적	자유롭게 라우팅 가능
재검색·재작성	구현 복잡	노드로 자연스럽게 표현
Self-RAG·HyDE 등 고급 패턴	적용 어려움	적용 용이
적합 상황	단순 QA	품질 검증·재시도가 필요한 프로덕션

핵심 설계 원칙
•	청크 크기·오버랩을 도메인에 맞춰 조정함(코드·법률·논문 등 특성 반영).
•	임베딩 모델은 태스크·언어·비용을 고려하여 선택함.
•	메타데이터를 풍부하게 부여하여 필터링·리랭킹의 근거로 활용함.
•	검색 품질과 생성 품질을 분리 평가함(RAGAS 등).
벡터 DB 선택 가이드
DB	특징	적합 상황
Chroma	가벼움, 로컬 파일 저장	프로토타입, 소규모 앱
FAISS	인메모리, 초고속 검색	정적 데이터, 배치 검색
Pinecone	매니지드, 자동 확장	프로덕션 SaaS
Weaviate / Qdrant	오픈소스, 하이브리드 검색	온프레미스 프로덕션
pgvector	PostgreSQL 확장	기존 RDB와 통합 관리

 3-2. 문서 처리 및 벡터 데이터베이스
문서 로딩
•	PyPDFLoader: PDF 파일을 페이지 단위로 로드함.
•	TextLoader: 텍스트·마크다운 파일을 단일 Document로 로드함.
•	WebBaseLoader: 웹 페이지를 크롤링하여 로드함.
•	각 Document는 page_content(본문)와 metadata(source, page 등)로 구성됨.
문서 청킹 전략
•	RecursiveCharacterTextSplitter가 표준 — 문단·문장·단어 순으로 재귀 분할함.
•	chunk_size: 임베딩·컨텍스트 창을 고려하여 300~1000자 수준에서 조정함.
•	chunk_overlap: 문맥 손실 방지를 위해 청크의 10~20%를 겹침으로 설정함.
•	코드·표·목록은 구조 보존 위해 특수 splitter(예: MarkdownHeaderTextSplitter) 활용.
임베딩과 벡터 DB
•	임베딩: 텍스트를 고차원 벡터로 변환하여 의미적 유사도 계산을 가능케 함.
•	OpenAI text-embedding-3-small/large가 성능·비용 균형에서 표준적 선택임.
•	Chroma: from_documents로 임베딩+저장을 한 번에 처리 가능. 재로드 시 embedding_function 필수.
[예시] Chroma 벡터 DB 생성
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectordb = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",
)


Retriever 패턴
•	as_retriever(search_kwargs={"k": 3}) 형태로 벡터 스토어를 검색 인터페이스로 노출함.
•	search_type="mmr"로 Maximal Marginal Relevance 검색 지정 시, 유사도와 다양성을 동시에 확보함.
•	similarity_search_with_score로 점수 기반 임계값 필터링 가능함.
검색 최적화 기법
기법	설명	효과
쿼리 확장	LLM으로 원 질문을 여러 형태로 변형	리콜 향상
하이브리드 검색	Dense(벡터) + Sparse(BM25) 앙상블	정확한 키워드+의미 결합
메타데이터 필터	source, page 등으로 검색 범위 제한	정밀도·속도 개선
리랭킹 (Rerank)	Cross-encoder로 상위 결과 재정렬	상위 K 품질 상승

 3-3. StateGraph 기반 RAG 워크플로우
기본 RAG: Retrieve → Generate
가장 단순한 형태는 검색 노드(retrieve)와 생성 노드(generate) 두 개로 구성됨. State는 question, documents, answer 등을 포함함. LCEL 대비 이점은 조건부 분기·재검색을 자연스럽게 확장 가능하다는 점임.
[예시] 기본 StateGraph RAG
class RAGState(TypedDict):
    question: str
    documents: list
    answer: str

def retrieve(state):
    docs = retriever.invoke(state["question"])
    return {"documents": docs}

def generate(state):
    ctx = "\n\n".join(d.page_content for d in state["documents"])
    ans = llm.invoke(f"질문:{state['question']}\n컨텍스트:{ctx}").content
    return {"answer": ans}

g = StateGraph(RAGState)
g.add_node("retrieve", retrieve)
g.add_node("generate", generate)
g.add_edge(START, "retrieve")
g.add_edge("retrieve", "generate")
g.add_edge("generate", END)


조건부 재검색 패턴
•	검색 결과의 유사도 점수가 임계 미만이면 쿼리를 재작성 후 재검색함.
•	생성된 답변에 근거 부족(‘모른다’) 신호가 있으면 다른 소스로 재시도함.
•	라우터 노드에서 결정하고, 최대 재시도 횟수를 State에 관리하여 무한 루프 방지.
고급 RAG 패턴 비교
패턴	핵심 아이디어	장점
Multi-Query	원 질문을 여러 하위 질문으로 확장 후 검색 결과 병합	리콜 향상
HyDE	가상의 답변을 생성해 그 임베딩으로 검색	쿼리-문서 임베딩 간극 완화
Self-RAG	검색 필요성·근거 충분성·답변 품질을 스스로 판단·재시도	품질 최상, 복잡도↑
CRAG	검색 결과의 관련성 판정 후 웹 검색·재작성 등 보정	잘못된 검색 결과 방어

RAG + ReAct 에이전트 통합
RAG 파이프라인 자체를 knowledge_search 도구로 래핑하여, create_agent에 다른 도구들과 함께 등록함. 에이전트가 상황에 따라 문서 검색·계산·웹 검색 도구를 자율적으로 선택함.
[예시] RAG를 도구로 래핑
@tool
def knowledge_search(query: str) -> str:
    """사내 문서에서 관련 정보를 검색함."""
    docs = retriever.invoke(query)
    return "\n\n".join(d.page_content for d in docs[:3])

agent = create_agent(
    model,
    tools=[knowledge_search, tavily_search, calculator],
    checkpointer=InMemorySaver(),
)


 3-4. 평가 및 최적화
평가가 중요한 이유
RAG 시스템은 검색·생성 두 단계의 오류가 누적되므로, 정량 평가 없이는 품질 개선이 불가능함. 정성 평가만으로는 회귀·A/B 비교가 어렵기 때문에, 자동화된 메트릭 파이프라인이 필수적임.
주요 평가 메트릭 (RAGAS 기준)
메트릭	측정 대상	의미
Faithfulness	답변이 컨텍스트에 근거하는가	환각 억제 수준
Answer Relevancy	답변이 질문에 부합하는가	질문-답 일치도
Context Precision	검색된 컨텍스트 중 관련 있는 비율	검색 노이즈 수준
Context Recall	정답에 필요한 정보를 얼마나 회수했는가	검색 리콜

성능 최적화 방향
•	컨텍스트 최적화: 유사도 임계값 필터링, 상위 K 조정, MMR로 다양성 확보.
•	캐싱: 동일·유사 질문에 대한 임베딩·검색·응답 결과 캐시로 지연·비용 절감.
•	인덱스 선택: 데이터 규모에 맞춰 Flat / IVF / HNSW 등 인덱스 유형 조정.
•	리랭킹: cross-encoder를 상위 결과에만 적용하여 정밀도 강화.
Part 3 마무리
RAG는 ‘검색 품질 = 생성 품질’의 근사에 가깝다. 청킹·임베딩·검색 최적화의 세 축을 정량 평가로 지속 개선함으로써 신뢰할 수 있는 지식 기반 시스템을 구축 가능함. StateGraph는 이 반복 개선을 표현하는 최적의 도구임.

 Part 4. 다중 에이전트 시스템
본 파트는 여러 에이전트가 역할을 분담하여 협업하는 다중 에이전트 시스템의 설계와 구현을 다룸. 단일 에이전트로 다루기 어려운 복잡 업무를 분할·조율함으로써 정확도와 확장성을 확보함.
4-1. 다중 에이전트 시스템 개요와 아키텍처
다중 에이전트가 필요한 이유
•	역할 분리: 전문화된 프롬프트·도구·모델을 각 에이전트에 할당하여 성능 극대화.
•	컨텍스트 관리: 각 에이전트가 필요한 정보만 다루므로 프롬프트 길이·비용 절감.
•	확장성: 새 도메인이 추가되면 새로운 워커 에이전트만 붙이면 됨.
•	유지보수성: 에이전트 단위 테스트·교체가 용이함.
세 가지 핵심 패턴
패턴	구조	특징
Subagents (Supervisor)	상위 Supervisor가 하위 전문 워커를 도구처럼 호출	중앙 통제, 라우팅 명확
Handoffs	에이전트가 대화 흐름의 소유권을 다른 에이전트로 이양	단계별 대화형 워크플로우
StateGraph 커스텀	공유 State 위에 에이전트 노드들을 직접 배치	복잡한 조율·병렬 가능

패턴 선택 가이드
•	명확한 중앙 조정자·계층 구조가 필요하면 Supervisor.
•	단계에 따라 전문 담당자가 바뀌는 대화형 프로세스는 Handoffs.
•	병렬·복합 워크플로우, 자유로운 상태 공유가 필요하면 StateGraph 커스텀.
4-1-2. 에이전트 간 통신 메커니즘
•	공유 State: AgentState 확장으로 여러 에이전트가 동일 State에 읽기·쓰기 수행.
•	메시지 기반: HumanMessage/AIMessage/ToolMessage 형태로 표준 스트림 활용.
•	Command 객체: 노드가 update+goto를 반환하여 라우팅과 상태 전달을 통합.
•	@tool 래핑: 에이전트를 도구로 감싸 상위 에이전트에서 함수처럼 호출함.
•	InMemorySaver + thread_id: 다중 에이전트 대화 히스토리를 세션 단위로 관리.
결과 접근 팁
에이전트 호출 결과에서 최종 답변은 .content가 아닌 .text 속성으로 접근해야 하는 경우가 있음. 메시지 종류별 접근 방법을 문서에서 확인 후 사용함.

 4-2. Subagents & Handoffs 패턴 구현
4-2-1. Subagents (Supervisor) 패턴
Supervisor 에이전트가 사용자의 요청을 이해하고, 적절한 워커 에이전트(웹 검색·Wikipedia·사내 문서 등)에 작업을 위임하는 계층 구조임. 워커는 도구로 래핑되어 Supervisor의 도구 목록에 포함됨.
구현 4단계
•	1단계 — 도구 초기화: TavilySearch, WikipediaQueryRun 등 원천 도구 준비.
•	2단계 — 워커 에이전트 생성: 각 도구에 특화된 create_agent 인스턴스 생성.
•	3단계 — 도구 래핑: 워커 에이전트를 다시 @tool 함수로 감싸 Supervisor가 호출 가능하게 함.
•	4단계 — Supervisor 생성: 래핑된 도구들을 Supervisor의 tools에 등록하여 최종 완성.
[예시] Supervisor 패턴 최소 구현
web_agent  = create_agent(model, [tavily_search],
                          system_prompt="웹 검색 전문가")
wiki_agent = create_agent(model, [wikipedia_query],
                          system_prompt="Wikipedia 전문가")

@tool
def call_web(query: str) -> str:
    """웹 검색으로 최신 정보를 찾음."""
    return web_agent.invoke({"messages":[("user", query)]})["messages"][-1].content

@tool
def call_wiki(query: str) -> str:
    """Wikipedia로 배경 지식을 찾음."""
    return wiki_agent.invoke({"messages":[("user", query)]})["messages"][-1].content

supervisor = create_agent(model, [call_web, call_wiki],
                          system_prompt="여러 리서치 도구를 조율하는 관리자")


장점과 한계
•	장점: 라우팅 로직이 명확, 워커 추가·교체 용이, 각 워커가 자체 컨텍스트 유지.
•	한계: Supervisor의 판단이 병목이 될 수 있음, 워커 간 직접 협업은 어려움.
4-2-2. Handoffs 패턴
Handoffs는 대화의 소유권이 단계에 따라 다른 전문 에이전트로 이양되는 패턴임. 고객 지원(문의 접수 → 문제 분류 → 해결 안내) 등 단계형 프로세스에 적합함.
구현 4단계
•	1단계 — 상태 정의: 현재 단계(step)와 수집 데이터를 담는 커스텀 State 설계.
•	2단계 — 전환 도구 정의: transfer_to_next_stage(...) 형태의 @tool로 단계 이양 명령 표현.
•	3단계 — 미들웨어 구성: 단계별 프롬프트·도구·모델을 동적으로 스왑함.
•	4단계 — 에이전트 생성 및 실행: 단일 create_agent를 미들웨어와 함께 사용해 멀티 턴 대화 처리.
사례: 3단계 고객 지원 흐름
Turn	담당 에이전트	역할
1	warranty_collector	보증 ID를 요청·수집함
2	issue_classifier	문제 유형(HW/SW)을 분류함
3	resolution_specialist	해결 방안(무상 수리 등)을 안내함

Subagents vs Handoffs
기준	Subagents (Supervisor)	Handoffs
구조	계층적 (1:N)	순차적 이양 (선형/조건부)
소유권	Supervisor가 항상 유지	단계 담당자로 이동
대표 사례	리서치·다도구 오케스트레이션	고객 지원, 인테이크 프로세스
복잡도	라우팅 로직 중심	상태 전이·미들웨어 중심

다중 에이전트 실무 팁
패턴은 배타적이지 않음. 실무에서는 Supervisor 아래에 Handoffs 서브플로우를 두거나, StateGraph 위에 두 패턴을 조합하는 하이브리드 설계가 흔함. 요구사항의 ‘단계성 vs 병렬성’을 먼저 판단한 후 패턴을 선택함.

마무리
LangGraph의 강점은 ‘명시성’에 있음. 상태·흐름·인간 개입 지점을 코드로 명확히 드러내므로, 복잡해질수록 오히려 통제 가능성이 높아짐. 본 자료의 개념들을 순환적으로 재학습하면서 실제 프로젝트에서 조금씩 확장 적용해 나가면, 견고한 LLM 시스템 설계 역량을 자연스럽게 체득할 수 있음.
